# なぜ今モノレポが人気なのか

> この文書は Turborepo 導入判断の検討メモとしてまとめたもの。

## この文書の位置づけ

「初期設定が重いなら、そこまでしてまとめる必要ある？」という疑問への補足資料です。

この文書は導入を推すためではなく、なぜ最近モノレポが選ばれやすくなったのかを客観的に整理するために置いています。

## 結論

最近モノレポが支持されているのは、単なる流行ではなく、**複数アプリと共有資産を同時に扱う開発が増えたこと** と、**それを支えるツールチェーンが実務で使える水準まで成熟したこと** が大きな理由です。

以前は「大規模企業向けの特殊運用」と見なされがちでしたが、現在は pnpm workspaces、Turborepo、Nx、Changesets、Docker Compose、GitHub Actions などの選択肢が整っています。これにより、小規模チームでも条件が合えば現実的な選択肢になります。

---

## 人気の背景

### 1. フルスタックで同時に変える要求が増えた

現在のプロダクト開発は、web のみ、api のみで完結しにくくなっています。

- web
- api
- mobile
- shared UI
- shared config
- 契約パッケージ
- codegen 生成物

こうした構成では、1 つの機能追加でも複数レイヤーを横断します。モノレポであれば、1 つの PR で原子的に変更をそろえやすくなります。

### 2. OpenAPI や GraphQL など契約駆動が広がった

API 契約を中心に型やクライアントを生成する開発では、契約と consumer のズレが運用コストになります。

モノレポは、契約、生成物、利用側アプリを同じ変更セットで更新しやすく、この点は支持が広がる主要因の 1 つです。

### 3. 共通設定の標準化需要が高くなった

TypeScript、ESLint、Prettier、Jest、Vitest、Storybook、stylelint など、現代のフロントエンドや TypeScript backend では共有したい設定が増えています。

repo が分かれていると、設定差分と更新漏れが負債化しやすくなります。モノレポでは標準設定を 1 箇所に集約しやすくなります。

### 4. CI と cache の重要性が増した

リポジトリが分かれていると、横断変更のたびに複数 pipeline をまたぐ必要があります。モノレポ + Turbo 系構成では、変更影響に限定して CI を実行しやすくなります。

つまり、最近の支持は「1 repo に集約できて便利」という理由だけでなく、**CI の無駄を減らせる** こととも強く結びついています。

### 5. ツール成熟で現実的になった

以前のモノレポは、独自スクリプトや重いビルド基盤を必要とし、再現性の確保が難しい面がありました。

現在は、次のような構成が一般化しています。

- pnpm workspaces: 厳格な依存解決と速い install
- Turborepo / Nx: タスク依存、キャッシュ、影響範囲実行
- Changesets: versioning と publish
- Docker Compose: ローカル実行環境の統一
- GitHub Actions: monorepo 前提の CI 実装

この成熟により、モノレポは「理想論」ではなく「実務上の選択肢」へと移行しました。

### 6. AI エージェント活用との相性が高い

AI エージェントを開発フローに組み込むケースが増える中で、コード探索・依存追跡・影響範囲特定を一貫して行える構造の価値が上がっています。

モノレポでは、関連するアプリ、ライブラリ、設定、契約定義が同一リポジトリ内にあるため、AI が変更文脈を横断的に把握しやすくなります。結果として、次の効果が期待できます。

- 仕様変更時の影響調査の高速化
- 修正提案の精度向上
- 変更漏れの抑制

特に複数リポジトリをまたぐ構成と比較すると、コンテキスト分断による探索コストを下げやすい点が実務上の利点です。

### 7. 設計されたモノレポは見通しを改善しやすい

Google の『Software Engineering at Google』で示されているように、モノレポは「集約そのもの」が価値なのではなく、依存関係・所有境界・変更手順を設計し、継続的に保守することで効果が出ます。

適切に設計されたモノレポでは、次の点で見通しを確保しやすくなります。

- 依存関係が明示され、変更影響を追跡しやすい
- 共通ルールが統一され、判断基準がぶれにくい
- 横断変更の手順が標準化され、運用が属人化しにくい

一方で、境界設計や運用ルールが曖昧なまま集約すると、逆に複雑性が増すため、導入時にはガバナンス設計が前提になります。

## それでも人気が出るほど万能なのか

ただし、支持されていることと、すべての組織に適していることは別問題です。

### 向いているケース

- 小さくても複数 app を同時に育てる
- shared package が早い段階で必要
- API 契約や codegen を中心に回す
- 設定や品質ゲートを標準化したい

### 向いていないケース

- 単一 app で完結する
- チームごとの独立性が強く、同時変更も少ない
- release cadence や権限モデルが完全に別
- infra や pipeline をまとめるより分けたほうが明確

---

## 「最近人気」をどう説明すると自然か

次の 4 点で要旨を説明できます。

1. 複数 app と shared package を同時に扱う開発が増えた
2. 契約駆動や codegen の普及で、横断変更を 1 つの変更セットで扱う価値が上がった
3. pnpm や Turborepo の成熟で、小規模チームでも現実的に回せるようになった
4. AI エージェント活用と整合しやすく、適切に設計すれば見通しを維持しやすい

---

## 誤解しやすい点

- 人気だから正解、ではない
- モノレポにしただけでは速くならない
- 依存管理の健全化は pnpm などの責務で、Turbo 単体の効果ではない
- 開発サーバの常駐問題は Compose などとの責務分離が必要なことがある

---

## まとめ

最近モノレポが支持されているのは、**複雑なアプリ群を横断保守する需要** と、**それを支えるツールの成熟** が重なったためです。加えて、AI エージェント活用が進む中で、適切に設計されたモノレポは変更影響の見通しを維持しやすい点も背景にあります。

したがって導入判断では、「モノレポが人気か」より「自分たちの repo で横断変更、共有設定、契約駆動、CI 高速化、AI エージェント活用の必要性がどれだけ強いか」を評価することが重要です。
