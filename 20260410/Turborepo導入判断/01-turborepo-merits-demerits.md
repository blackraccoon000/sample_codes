# Turborepo のメリットとデメリット

> この文書は Turborepo 導入判断の検討メモとしてまとめたもの。

## 先に短く答える

「初期設定が煩雑で、そこまでして 1 つにまとめる必要があるか」という疑問に対しては、次の答えが実務的です。

- 単一アプリ寄りなら無理に導入しなくてよい
- 複数パッケージで順序制御や codegen 同期が必要なら導入価値が高い

つまり Turborepo は、最初から必須の道具ではなく、複雑さが増えてきたときに効く道具です。

## 前提

最初に切り分けるべきなのは、**モノレポや workspaces が解決すること** と、**Turborepo が追加で解決すること** は別だという点です。

- モノレポ / workspaces: 依存管理、内部パッケージのリンク、共通設定の集約
- Turborepo: タスクの依存関係、並列実行、キャッシュ、影響範囲の絞り込み

この切り分けが曖昧だと、Turborepo の評価軸がぶれやすくなります。

## メリット

| 観点                           | 内容                                                                   | 実務で効く場面                                                             |
| ------------------------------ | ---------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| タスク順序の制御               | generate、lint、typecheck、test、build などを依存順で回せる            | OpenAPI 変更後に generated client と型を先に揃えてから各アプリを検証したい |
| キャッシュ                     | 変更がないタスクを再実行せずに済む                                     | CI の build や typecheck を短縮したい                                      |
| 影響範囲の限定                 | 変わったパッケージとその依存先だけを対象にできる                       | 大きめのモノレポで毎回全件実行したくない                                   |
| 共通コマンドの入口統一         | generate、lint、test、typecheck などを repo ルートから揃えて実行できる | チーム内で実行手順を揃えたい                                               |
| OpenAPI との相性               | スキーマ更新から生成物の反映、検証までを一連の依存グラフに乗せやすい   | 契約駆動で API とクライアントを同時に進めたい                              |
| パッケージマネージャ非依存     | npm、pnpm、yarn などの上に載せられる                                   | 既存構成を大きく壊さずに導入判断したい                                     |
| Compose との責務分離がしやすい | 実行環境は Compose、コード内タスクは Turbo と分けられる                | 開発サーバ、DB、キュー、モック API が混在する                              |

## 判断材料として有効な論点

### 1. OpenAPI を正本にした同期的な setup

この論点は実務上の効果が大きいです。別リポジトリでも実現は可能ですが、実際には次の調整が増えます。

- スキーマ変更の配布
- 生成物の再配布
- 利用側リポジトリでの取り込み
- 破壊的変更の検知と移行タイミングの調整

モノレポ + Turborepo であれば、同じ変更セット内で generate と consumer 側検証まで一貫して実行できます。結果として、契約駆動開発の運用コストを下げやすくなります。

### 2. 共通設定の集約を運用可能にする

TypeScript、ESLint、Prettier の共通化自体は workspace だけでも可能ですが、変更後の typecheck や lint を継続的に高速実行できるかは別の課題です。

共通設定を置くだけでなく、変更影響を repo 全体で安全に確認しやすくするところまで含めると、Turborepo の価値が明確になります。

### 3. Docker Compose との棲み分け

Turborepo の persistent task は長時間動作する性質上、他タスクの依存先にしにくい特徴があります。常駐サーバを依存チェーンの中核に置くと、後続が終了待ちで進まなくなるためです。

そのため、次の分担は実装・運用の両面で妥当です。

- Docker Compose: API、DB、Redis、モックサーバ、開発コンテナなどの実行環境
- Turborepo: generate、lint、typecheck、test、build、setup の依存関係

これは「Turborepo の弱点回避」というより、責務に応じた適切な分離設計です。

## デメリット

| 観点                             | 内容                                                                | 補足                                                          |
| -------------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------- |
| 初期設計が必要                   | turbo.json、package scripts、outputs、inputs の設計が要る           | 雑に入れると効果が薄い                                        |
| 学習コスト                       | cache、persistent、dependsOn、filter の理解が必要                   | チーム全員が最低限の前提を持つ必要がある                      |
| 常駐タスクの制約                 | persistent task を依存先にできない                                  | dev サーバ連携は Compose や別 setup task で分離するのが現実的 |
| 小規模では過剰                   | パッケージが少ないと、設定の重さが勝ちやすい                        | 単一 app なら workspace + npm scripts でも足りる              |
| キャッシュ設計の落とし穴         | inputs、outputs、env の定義が粗いと誤キャッシュや無駄再実行が起きる | 特に codegen 周りは慎重に定義したい                           |
| 導入しても依存管理問題は消えない | 依存解決は pnpm などの仕事で、Turbo 自体は面倒を見ない              | ファントム依存対策は pnpm 側の責務                            |

## メリデメ表

| 項目                         | 評価           | 理由                                                                          |
| ---------------------------- | -------------- | ----------------------------------------------------------------------------- |
| 共通設定の一元化             | 条件付きで強い | 集約そのものは workspace の役割だが、その後の検証を高速に回せる点で価値が増す |
| OpenAPI 生成フロー           | とても強い     | schema 変更から consumer 検証までを単一リポジトリで依存順に回せる             |
| CI 高速化                    | 強い           | キャッシュと影響範囲の絞り込みが効く                                          |
| ローカル DX                  | 強い           | ルートコマンドで generate、lint、test、typecheck を揃えやすい                 |
| 小規模構成への適合           | 弱い           | package 数が少ないと投資対効果が薄い                                          |
| 長寿命 dev server の取り回し | 弱みあり       | persistent task 制約があるため、Compose などとの併用設計が必要                |
| 依存関係の健全化             | 限定的         | ここは pnpm workspaces 側の貢献が大きい                                       |

---

## 導入を勧めやすい条件

1. アプリやパッケージが複数ある
2. 共有設定や内部パッケージが増える見込みがある
3. OpenAPI や codegen を repo 内で同期管理したい
4. CI 時間が伸びてきている
5. 変更影響を絞って検証したい

## まだ早い条件

1. 実質 single app に近い
2. 共有パッケージがほとんどない
3. codegen がない
4. CI がまだ軽い
5. チームが turbo.json の運用を覚えるコストのほうが重い

---

## ひとことで言うなら

Turborepo は「モノレポを作るための道具」というより、**モノレポを実務速度で運用し続けるための道具** と捉えると理解しやすくなります。
